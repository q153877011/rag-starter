七、技术选型：开发者如何判断一个项目 / 工具是否靠谱
与之前的创业者受访者（ Daniel 让  Claude Code sub-agent 做  research ）不同， Tony 的技术选型方式很传统 —— 不
用  AI 做评估，而是依赖经验和人工检查。他看的指标是  repo 本身的健康度，而非  AI 的推荐。
评估维度 具体做法
NPM 下载量 去  NPM 看周下载量 ——500 下载  = 不靠谱，百万级  = 可信任
GitHub Stars看  stars 数量，作为社区认可度的快速指标
活跃度 看最近一次  commit 时间 —— 应该是最近几天内；项目活跃意味着有人在维护，出问题能修
贡献者背景查主要贡献者的  LinkedIn / GitHub 背景，看他们是否有相关领域经验，是否 "legit" （靠谱）
包体积 看依赖包大小 —— 避免为小功能引入巨大依赖（前端尤其敏感，影响加载速度）
内部替代 优先选公司已经在用的工具，即使不是  100% 完美 —— 已被审查过的工具更安全
"I would go to NPM and look at downloads. Go to GitHub and look at stars. Look at whether a project is being
actively maintained. Look up the major contributors' LinkedIn or GitHub profiles to see if they are legit."
「去  NPM 看下载量， GitHub 看  stars ，看项目是否活跃，查贡献者的  LinkedIn 看是否靠谱。」
Vercel 有  IT 维护的合规工具清单，新工具需过安全审查。 OpenClaw （安全风险）已从公司电脑封禁，其余通过企业
协议使用。 Notion MCP 曾因安全审查被  IT 短暂禁用，但因内部需求太大， IT 最终 " 走回头路 " 允许使用 —— 这说明开
发者的使用需求在某些情况下可以倒逼安全策略调整。
七、 Vercel 的开发者营销（ Developer Experience 内部视角）
作为开发者工具公司的内部工程师， Tony 提供了一个从内部看  Vercel 如何做  developer experience （简称 DX ）的视
角。
核心产品魔法： Git Push → 自动部署。 Tony 认为  Vercel 最成功的一点是极大降低了 " 改代码 " 到 " 看到效果 " 之间的
摩擦—— 把  Git 仓库和网站绑定，每次推代码就自动构建部署，不喜欢就像  Git 回滚一样回退。所有其他开发者体验
都建立在这个简单而强大的基础之上。
"There's something really magical about it — you push code, and the website changes. It drastically reduces th
e friction."
「有种魔法感 —— 你推代码，网站就变了。大幅减少了摩擦。」
DX 设计哲学： " 一条命令搞定 " 。 作为开发者给开发者做产品， Vercel 团队天然理解什么样的体验是 " 好 " 的 —— 完成
一件事应该只要一条命令，而不是一个巨大的配置文件。命令行工具、 Dashboard 、 GitHub 集成都围绕这个原则设
计。
"If you want to get something done, it should be just one command, instead of a giant config file."
「做一件事应该只要一条命令，而不是一个巨大的配置文件。」
